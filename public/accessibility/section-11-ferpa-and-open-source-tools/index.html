    Section 11. FERPA and open-source local tools¶

    Toolkit

    Home · Examples · Checker · Routing Guide

    Accessibility work is not just a Title II concern. Whenever you process student work — captioning a recording, transcribing audio, generating alt text — student data is involved. FERPA applies, and FERPA does not pause for accessibility deadlines. This section names the tension and lists the local-only / open-source tools that solve it.

    The tension¶

    If you upload a student-submitted video to a free cloud captioning service to generate captions, you have just transmitted student educational data to a third party. That third-party may log it, train models on it, or retain it. That is a FERPA violation regardless of how accessible the captions are.

    The fix is not to skip captions. The fix is to use tools that run locally on your computer so the file never leaves your machine.

    Disclaimer¶

    The list below is informational. Use at your own discretion and verify each tool against your own institution's data-handling policies before processing student records. Nothing here constitutes legal advice. If a student's data is exposed because you used a cloud tool when a local one would have done, that is your responsibility, not mine.

    Open-source local tools (run on your machine, no data leaves)¶

    Speech-to-text and captioning¶

    • OpenAI Whisper — open-source speech recognition model, runs locally on Mac/Linux/Windows. The reference implementation. Multi-language. github.com/openai/whisper
    • Vibe Transcribe — free desktop app wrapping Whisper. Drag-drop interface. Mac/Windows/Linux. Completely offline. thewh1teagle.github.io/vibe
    • MacWhisper — Mac-only Whisper GUI. Free tier sufficient for most use; paid tier for batch and longer audio. Local-only.
    • Subtitle Edit — free, open-source, Windows-focused subtitle editor with speech-recognition integration. Apache 2.0.
    • CCExtractor — open-source caption extraction from video files (DVB, ATSC, DVD, etc.). The standard for working with already-captioned video. GPL-2.0. ccextractor.org

    Transcript editing¶

    • oTranscribe — free, browser-based, MIT-licensed transcript editor. Audio stays on your computer (does not upload). otranscribe.com
    • Subtitld — open-source subtitle/caption editor for Linux/Mac/Windows. GPL-3.0.
    • transcribee — open-source automated draft + manual editing tool. AGPL-3.0. github.com/transcribee/transcribee

    Image accessibility (alt text, OCR)¶

    • Tesseract OCR — Apache-2.0 open-source OCR engine. Convert images of text into text. Local. github.com/tesseract-ocr/tesseract
    • OCRmyPDF — wraps Tesseract for PDF processing. Adds an OCR text layer to scanned PDFs. Free, MPL-2.0.

    PDF accessibility¶

    • PAC 2024 (PDF Accessibility Checker) — free, runs locally on Windows. The standard for testing PDF/UA conformance. pac.pdf-accessibility.foundation
    • OCRmyPDF — see above; adds searchable text layer to scanned PDFs.

    Document accessibility checking¶

    • WAVE Browser Extension — free in-browser accessibility checker (web pages only). Runs locally; no data sent to WebAIM. wave.webaim.org/extension
    • NVDA — free open-source screen reader for Windows. Test what your students will hear. GPL-2.0. nvaccess.org
    • VoiceOver — built into macOS. Already on your Mac. Cmd+F5 to enable.

    Practical workflows¶

    Caption a Zoom recording without sending student data anywhere:

    1. Download the local recording (Zoom → Recordings → "Download to local").
    2. Open Vibe Transcribe (or MacWhisper) → drop the video file in.
    3. Wait. The model runs on your machine; nothing uploads.
    4. Export .vtt or .srt.
    5. Upload only the caption file to wherever the video lives.

    OCR a scanned PDF without uploading the document:

    1. Open Terminal.
    2. brew install ocrmypdf (Mac) or your platform's installer.
    3. ocrmypdf input.pdf output.pdf — adds a searchable text layer locally.

    Alt-text suggestions without sending the image to the cloud:

    There is no equivalent of cloud AI vision running locally that's as good. The pragmatic approach is to write alt text by hand — it takes 30 seconds per image and is the only way to guarantee no data leaves your machine. If a student-facing image is in a document students submit (not one you wrote), you should not be uploading it to any cloud service for alt text generation regardless.

    What about commercial tools?¶

    Commercial cloud services (Otter.ai, Rev, Trint, Descript, Sonix, etc.) are popular and well-engineered, but they all transmit your file to their servers. For materials that contain student records, that is a FERPA risk. For your own lectures with no student information, commercial cloud services may be fine — verify each provider's data-handling policy and any institutional approvals before sending sensitive content.

    When in doubt: default to the local tool. Local-only is always FERPA-compliant by construction.

    A short workflow heuristic¶

    For any accessibility task involving content that includes or could include student data:

    1. Is this content student work or contains student information? If yes, use a local tool only. Treat any cloud option as out of bounds.
    2. Is this my own lecture / handout / syllabus with no student names or work in it? Either local or cloud is fine, but local is always safe. Cloud may be faster.
    3. Am I unsure? Default to local. There is no penalty for using Whisper to caption a non-FERPA file. There is a real penalty for sending FERPA data to a cloud service.

    If in doubt about whether a tool is appropriate: ask UMaine's Office of General Counsel or your ADA Coordinator before processing the content.
