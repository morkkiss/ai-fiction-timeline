Each example below shows a real course-page situation in two versions: inaccessible (what most existing course content looks like) and accessible (the same content with the fixes applied). Below each pair is a plain-language list of what changed and why, plus copy-paste-ready source code for the accessible version.

Course Page

Headings, alt text, link text, captions

This is the most common kind of page you build for a class. The 'before' has every default mistake faculty make. The 'after' is what an accessible course page looks like — same content, fixed.

→ Read the Field Guide section that explains this (Section 2 — The three things that matter most)

✗ Inaccessible — what most pages look like now
✓ Accessible — same content, fixed

What changed and why

  • Real headings instead of bold text
    Bold text that LOOKS like a heading is invisible to screen readers. The fixed page uses real <h1> and <h2> tags. In Word: select the text, click Heading 1 in Styles. In Google Docs: Format → Paragraph styles → Heading 1.
  • Image gets a real description
    The 'before' image has no alt text, so screen reader users hear nothing. The 'after' image describes what the picture shows: 'Five students sitting around a wooden seminar table sharing draft pages.'
  • Link text describes the destination
    'Click here' tells screen reader users nothing. 'Read the Week 3 Assignment Sheet (PDF)' tells them exactly where they're going.
  • Page has a real title
    The 'before' page has an empty <title>. Browsers show a useless tab name and screen readers announce nothing. The 'after' is titled 'ENG 307 — Week 3 — Saunders on Revision.'
  • Skip-to-content link for keyboard users
    Lets a keyboard-only user jump past navigation in one keystroke.
  • Video has a title and captions enabled
    The iframe has a descriptive title attribute, and the YouTube URL includes cc_load_policy=1 so captions show by default.
  • Page declares its language
    <html lang="en"> tells the screen reader to use English pronunciation.
View source — copy/paste-ready accessible code 
<!DOCTYPE html>
<html lang="en">
<head>
<meta charset="utf-8">
<title>ENG 307 — Week 3 — Saunders on Revision</title>
<style>body{font-family:Georgia,serif;max-width:640px;margin:1.5rem auto;padding:0 1rem;color:#222}h1{font-size:24px;margin-top:0;color:#1f4e79}h2{font-size:18px;margin-top:1.5rem;color:#1f4e79}a{color:#06c}img{max-width:100%}.skip{position:absolute;left:-9999px}.skip:focus{left:1rem;top:1rem;background:#fff;padding:.4rem .8rem;border:2px solid #1f4e79}</style>
</head>
<body>
<a class="skip" href="#main">Skip to main content</a>
<main id="main">
<h1>ENG 307 — Week 3</h1>
<h2>George Saunders on Revision</h2>
<p>This week we read a short essay by George Saunders on the iterative work of revision.</p>
<img src="https://placehold.co/400x250/e8d5c4/333?text=Workshop+Photo" alt="Five students sitting around a wooden seminar table sharing draft pages.">
<p>For this week's assignment, <a href="#">read the Week 3 Assignment Sheet (PDF)</a>.</p>
<h2>Watch before class</h2>
<iframe width="400" height="225" title="George Saunders on revision — eight-minute interview with captions" src="https://www.youtube.com/embed/dQw4w9WgXcQ?cc_load_policy=1" frameborder="0" allowfullscreen></iframe>
</main>
</body>
</html>

Equations (MathML)

Math that screen readers can actually read

Equation-heavy PDFs are the hardest accessibility problem on most courses. A picture of an equation is silent to a screen reader. MathML is structured math that the screen reader speaks aloud as math. The 'before' is the wrong way (image only). The 'after' is MathML — view source on the page to see the structure.

→ Read the Field Guide section that explains this (Section 5f — Math, science, and equations)

✗ Inaccessible — what most pages look like now
✓ Accessible — same content, fixed

What changed and why

  • Image of equation replaced with MathML
    The 'before' shows a picture of the quadratic formula. A blind student hears 'image' and nothing else. The 'after' uses MathML — the screen reader speaks 'x equals negative b plus or minus the square root of b squared minus four a c, all over two a.'
  • How to author MathML
    MathJax (free) lets you write LaTeX in your source and renders it as MathML in the browser. Microsoft Word's Equation Editor exports MathML when you save as MathML or HTML. Pandoc converts LaTeX documents to HTML+MathML.
  • If MathML is not possible
    Provide an alternative: a typed text description of the equation, an HTML version alongside the PDF, or a structured 'reading' of the formula using descriptive prose.
  • Avoid screenshotting equations
    MathType-rendered images, LaTeX-rendered PDFs, or screenshots of equations all fail the same way: they are pictures, not math. Treat any course with equations as needing a deliberate accessibility plan.
View source — copy/paste-ready accessible code 
<!DOCTYPE html>
<html lang="en">
<head>
<meta charset="utf-8">
<title>Quadratic formula example</title>
<style>body{font-family:Georgia,serif;max-width:600px;margin:1.5rem auto;padding:0 1rem}math{font-size:1.5em}</style>
</head>
<body>
<p>The quadratic formula is:</p>
<math display="block" xmlns="http://www.w3.org/1998/Math/MathML">
  <mi>x</mi>
  <mo>=</mo>
  <mfrac>
    <mrow>
      <mo>&#x2212;</mo><mi>b</mi>
      <mo>&#xB1;</mo>
      <msqrt>
        <msup><mi>b</mi><mn>2</mn></msup>
        <mo>&#x2212;</mo>
        <mn>4</mn><mi>a</mi><mi>c</mi>
      </msqrt>
    </mrow>
    <mrow><mn>2</mn><mi>a</mi></mrow>
  </mfrac>
</math>
<p>Use it to solve the problems on page 47.</p>
</body>
</html>

Forms

Labels, descriptions, error handling

An unlabeled form field is invisible. The screen reader user hears 'edit text, edit text, edit text' with no idea what to type in any of them. Labeling is one of the cheapest, highest-impact fixes you can make.

→ Read the Field Guide section that explains this (Section 5g — Forms and quizzes)

✗ Inaccessible — what most pages look like now
✓ Accessible — same content, fixed

What changed and why

  • Every input has a real <code>&lt;label&gt;</code>
    The 'before' has bare inputs. The 'after' wraps each input in a <label for="id"> with a matching id on the input. Screen readers announce the label as soon as the user lands on the input.
  • Required fields announce themselves
    required tells the browser and the screen reader that the field is mandatory.
  • Helper text uses <code>aria-describedby</code>
    The 'help' line under the quiz code is announced after the label, so users get the label, the requirement, and the constraint together.
  • Real button, real submit type
    <button type="submit"> is announced as a button. A bare div with an onclick handler is not.
  • Page heading sets context
    The form sits inside a page that has its own heading and title — so users know what form they're in before they start filling it.
View source — copy/paste-ready accessible code 
<!DOCTYPE html>
<html lang="en">
<head>
<meta charset="utf-8">
<title>Quiz access form</title>
<style>body{font-family:Georgia,serif;max-width:520px;margin:1.5rem auto;padding:0 1rem;color:#222}label{display:block;margin-top:.8rem;font-weight:600}input,select{display:block;margin-top:.25rem;padding:.45rem;width:100%;max-width:320px;border:1px solid #999;border-radius:3px}.help{font-size:.85em;color:#555;margin-top:.15rem}button{padding:.55rem 1.2rem;margin-top:1rem;background:#1f4e79;color:#fff;border:0;border-radius:3px;font-size:1em;cursor:pointer}</style>
</head>
<body>
<h1 style="font-size:1.2em;color:#1f4e79;margin-top:0">Quiz access form</h1>
<form>
  <label for="code">Quiz access code</label>
  <input id="code" name="code" type="text" required aria-describedby="code-help">
  <div id="code-help" class="help">Six characters, sent to your maine.edu email.</div>

  <label for="email">Your university email</label>
  <input id="email" name="email" type="email" required>

  <label for="attempt">Attempt number</label>
  <select id="attempt" name="attempt">
    <option value="1">First attempt</option>
    <option value="2">Second attempt</option>
    <option value="3">Third attempt</option>
  </select>

  <button type="submit">Begin quiz</button>
</form>
</body>
</html>

These examples illustrate the concepts in the Faculty Accessibility Field Guide. They are simplified for clarity — your real course materials will be longer and richer, but the same principles apply: real headings, real labels, real descriptions, real captions.
