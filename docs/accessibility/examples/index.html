    Working Examples¶

    See accessibility done right. Each example below shows the same content twice: an inaccessible version (what most existing course pages look like) and an accessible version (the same content with the fixes applied).

    The interactive side-by-side iframe version of these examples loads below. If the embed doesn't render (some browsers, some networks, some extensions block iframes), the same content is fully written out beneath the embed — examples, code, and explanations — so this page is useful regardless.

    Open the standalone interactive Examples page in a new tab

    Example 1 — Course page (inaccessible vs. accessible)¶

    The inaccessible version has every default mistake faculty make: an empty <title>, bold text passing as section headings, an image with no description, a "click here" link, an iframe video without a title, and no language declared.

    The accessible version uses real <h1> and <h2> tags via the Heading style menu in Word/Google Docs. Image alt text reads "Five students sitting around a wooden seminar table sharing draft pages." Link text reads "Read the Week 3 Assignment Sheet (PDF)" — a screen reader user knows where they're going. The page has a real title ("ENG 307 — Week 3 — Saunders on Revision"). A skip-to-main-content link helps keyboard users. The YouTube embed includes cc_load_policy=1 so captions show by default. <html lang="en"> tells the screen reader to use English pronunciation.

    Copy-paste-ready accessible code¶

    <!DOCTYPE html>
<html lang="en">
<head>
<meta charset="utf-8">
<title>ENG 307 — Week 3 — Saunders on Revision</title>
</head>
<body>
<a class="skip" href="#main">Skip to main content</a>
<main id="main">
<h1>ENG 307 — Week 3</h1>
<h2>George Saunders on Revision</h2>
<p>This week we read a short essay by George Saunders on the iterative work of revision.</p>
<img src="workshop.jpg" alt="Five students sitting around a wooden seminar table sharing draft pages.">
<p>For this week's assignment, <a href="week3-assignment.pdf">read the Week 3 Assignment Sheet (PDF)</a>.</p>
<h2>Watch before class</h2>
<iframe width="400" height="225" title="George Saunders on revision — eight-minute interview with captions"
        src="https://www.youtube.com/embed/VIDEO_ID?cc_load_policy=1" allowfullscreen></iframe>
</main>
</body>
</html>

    Example 2 — Equations rendered as MathML¶

    A picture of an equation is silent to a screen reader. The student hears "image" and nothing else.

    MathML is structured math that the screen reader speaks aloud as math. The quadratic formula authored in MathML is announced as "x equals negative b plus or minus the square root of b squared minus four a c, all over two a."

    How to author MathML:

    • MathJax (free) lets you write LaTeX in your source and renders it as MathML in the browser.
    • Microsoft Word's Equation Editor exports MathML when you save as MathML or HTML.
    • Pandoc converts LaTeX documents to HTML+MathML.

    If MathML isn't possible: provide an alternative — a typed text description of the equation, an HTML version alongside the PDF, or a structured "reading" of the formula in prose.

    Avoid screenshotting equations. MathType-rendered images, LaTeX-rendered PDFs, and screenshots all fail the same way: they are pictures, not math. Treat any course with equations as needing a deliberate accessibility plan.

    MathML source for the quadratic formula¶

    <math display="block" xmlns="http://www.w3.org/1998/Math/MathML">
  <mi>x</mi>
  <mo>=</mo>
  <mfrac>
    <mrow>
      <mo>&#x2212;</mo><mi>b</mi>
      <mo>&#xB1;</mo>
      <msqrt>
        <msup><mi>b</mi><mn>2</mn></msup>
        <mo>&#x2212;</mo>
        <mn>4</mn><mi>a</mi><mi>c</mi>
      </msqrt>
    </mrow>
    <mrow><mn>2</mn><mi>a</mi></mrow>
  </mfrac>
</math>

    Example 3 — Forms (labels, descriptions, error handling)¶

    An unlabeled form field is invisible. Screen reader users hear "edit text, edit text, edit text" with no idea what to type.

    The accessible version:

    • Every input has a real <label> programmatically associated with the field via for and id.
    • Required fields use the required attribute so they're announced as required.
    • Helper text uses aria-describedby so it's read after the label.
    • Buttons have visible text and type="submit" so they're announced as buttons.
    • Page heading sets context — users know what form they're in before filling it.

    Accessible form source¶

    <form>
  <label for="code">Quiz access code</label>
  <input id="code" name="code" type="text" required aria-describedby="code-help">
  <div id="code-help">Six characters, sent to your maine.edu email.</div>

  <label for="email">Your university email</label>
  <input id="email" name="email" type="email" required>

  <label for="attempt">Attempt number</label>
  <select id="attempt" name="attempt">
    <option value="1">First attempt</option>
    <option value="2">Second attempt</option>
    <option value="3">Third attempt</option>
  </select>

  <button type="submit">Begin quiz</button>
</form>
